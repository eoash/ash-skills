# 모노레포 vs 폴리레포

## 개념

- **모노레포(Monorepo)**: 여러 프로젝트를 하나의 Git 저장소에 모아두는 구조
- **폴리레포(Polyrepo)**: 프로젝트마다 별도 Git 저장소를 사용하는 구조

## 비교

| | 모노레포 | 폴리레포 |
|---|---|---|
| push | 한 번이면 끝 | 프로젝트마다 따로 |
| 코드 공유 | 같은 폴더니까 복붙/import 쉬움 | 패키지로 만들어야 공유 가능 |
| 커밋 히스토리 | 전체 프로젝트가 섞임 | 프로젝트별로 깔끔 |
| 배포 (Vercel) | Root Directory 지정 필요 | 레포 연결만 하면 끝 |
| 권한 관리 | 전체 레포 접근 or 없음 | 프로젝트별 세밀한 권한 가능 |
| CI/CD | 변경된 폴더만 빌드하도록 설정 필요 | 각 레포가 독립 빌드 |
| CLAUDE.md | 루트 공통 + 하위 전용 자동 병합 | 프로젝트마다 독립, 공통 규칙 중복 |

## 판단 기준

**모노레포가 맞는 경우:**
- 한 사람(또는 소수)이 여러 프로젝트를 넘나들며 작업
- 프로젝트 간 패턴/컴포넌트 공유가 잦음
- 공통 규칙을 한 곳에서 관리하고 싶을 때

**폴리레포가 맞아지는 시점:**
- 팀원 5명+ 이상이 각각 다른 프로젝트만 담당
- 프로젝트마다 배포 주기가 완전히 다를 때
- 외부 협력사에 특정 프로젝트만 열어줘야 할 때

## EO Studio 현황 (2026-03-08)

`eoash/eoash.git` 모노레포 안에 token-dashboard, finance-dashboard, leave-dashboard 등 운영.
token-dashboard는 **협업 가능성** 때문에 별도 레포(`eoash/token-dashboard`)도 유지 → 이중 push 필요.
모노레포는 개인 운영 전체(scripts, agent, 법무 등)가 들어있어서 외부 공유 불가.

## 혼합 전략 (모노레포 + 폴리레포)

둘 중 하나만 고를 필요 없음. **프로젝트 성격에 따라 혼합**:
- 혼자 쓰는 프로젝트 → 모노레포에만
- 협업할 프로젝트 → 모노레포 + 별도 레포 (동기화 스크립트로 관리)

## 대표 사례

Google, Meta, Uber, Twitter 등 대기업도 모노레포 사용.
핵심: "한 사람이 여러 프로젝트를 넘나드는가?" → Yes면 모노레포.
